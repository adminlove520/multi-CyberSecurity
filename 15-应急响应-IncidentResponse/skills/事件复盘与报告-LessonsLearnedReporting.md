# 📝 事件复盘与报告 (Lessons Learned & Reporting)

## 概述
事件复盘（Lessons Learned）是应急响应生命周期的最后阶段，也是最重要的改进环节。通过系统化地复盘事件处理的全过程，发现流程、技术和管理层面的短板，推动安全体系持续改进。参照 **NIST SP 800-61** 和 **SANS Incident Handler's Handbook** 的复盘方法论。

## 核心技能

### 1. 事件报告模板

```markdown
# 安全事件报告

## 1. 事件概要
| 项目 | 内容 |
|:---|:---|
| 事件编号 | IR-2024-0001 |
| 事件类型 | [勒索软件/数据泄露/APT/内部威胁] |
| 严重等级 | P1/P2/P3/P4 |
| 发现时间 | 2024-01-01 10:00 UTC+8 |
| 结束时间 | 2024-01-03 18:00 UTC+8 |
| 处置时长 | 56小时 |
| 报告人 | [姓名] |
| 报告日期 | 2024-01-04 |

## 2. 事件时间线
| 时间 | 事件 | 处置动作 | 负责人 |
|:---|:---|:---|:---:|
| T+0min | 发现异常告警 | 确认告警真实性 | SOC Analyst |
| T+15min | 确认P1事件 | 启动IR流程，组建IR团队 | IR Lead |
| T+30min | 完成初步评估 | 网络隔离+用户下线 | Network Team |
| T+2h | 完成内存取证 | 提取IoC列表 | Forensics Analyst |
| T+4h | 确认攻击类型 | 识别为某某攻击 | Threat Intel |
| T+8h | 遏制完成 | 所有失陷主机隔离 | IR Team |
| T+24h | 清除完成 | 重装+安全加固 | IT Operations |
| T+48h | 恢复完成 | 业务验证通过 | Business Owner |
| T+56h | 复盘会议 | 输出根本原因分析 | All Stakeholders |
```

### 2. 复盘会议议程

```text
=== 事件复盘会议议程 ===

时间: 事件解决后 3~5 天内
参与人: 
  - IR团队全体成员
  - IT运维负责人
  - 业务部门代表
  - 安全主管
  - （可选）法务/合规/PR

议程:
┌──────────────────────────────────────────────┐
│ 1. 事件回顾 (15min)                           │
│    ● 主持人介绍事件概况和时间线                  │
│    ● 确认事件事实无争议                         │
├──────────────────────────────────────────────┤
│ 2. 过程复盘 (30min)                           │
│    ● 按时间线逐段回顾处置过程                    │
│    ● 每个环节的响应时间、决策质量、执行效果         │
│    ● 哪些做得好？哪些可以改进？                   │
├──────────────────────────────────────────────┤
│ 3. 技术复盘 (30min)                           │
│    ● 攻击技术回顾 (TTP分析)                    │
│    ● 检测覆盖率评估 (哪些漏了？哪些准确？)         │
│    ● 工具效果评估 (EDR/SIEM/日志)              │
├──────────────────────────────────────────────┤
│ 4. 流程复盘 (20min)                           │
│    ● IR流程执行情况评估                         │
│    ● 沟通流程效率评估                           │
│    ● 外部协作评估 (法务/PR/执法)                │
├──────────────────────────────────────────────┤
│ 5. 行动项确认 (25min)                         │
│    ● 确定改进措施 (短期/中期/长期)              │
│    ● 分配责任人                              │
│    ● 设定完成期限和验收标准                      │
└──────────────────────────────────────────────┘
```

### 3. 改进措施跟踪

```sql
-- 改进措施跟踪表 (数据库设计)
CREATE TABLE ir_improvements (
    id INT PRIMARY KEY AUTO_INCREMENT,
    event_id VARCHAR(20) NOT NULL,        -- 关联事件编号
    category ENUM('技术', '流程', '人员', '工具', '其他') NOT NULL,
    description TEXT NOT NULL,             -- 改进措施描述
    priority ENUM('P0-紧急', 'P1-高', 'P2-中', 'P3-低') NOT NULL,
    owner VARCHAR(100) NOT NULL,          -- 负责人
    deadline DATE NOT NULL,               -- 完成期限
    status ENUM('待开始', '进行中', '已完成', '已关闭') DEFAULT '待开始',
    completion_note TEXT,                  -- 完成说明
    verified_by VARCHAR(100),              -- 验收人
    verified_date DATE,                    -- 验收日期
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 4. 事件度量指标

| 指标 | 定义 | 本次事件 | 行业基准 | 目标 |
|:---|:---|:---:|:---:|:---:|
| MTTD (Mean Time to Detect) | 平均发现时间 | 2小时 | 7小时 | <1小时 |
| MTTR (Mean Time to Respond) | 平均响应时间 | 15分钟 | 1小时 | <15分钟 |
| MTTC (Mean Time to Contain) | 平均遏制时间 | 30分钟 | 3小时 | <30分钟 |
| MTTER (Mean Time to Eradicate) | 平均清除时间 | 24小时 | 48小时 | <24小时 |
| MTTRC (Mean Time to Recover) | 平均恢复时间 | 48小时 | 72小时 | <24小时 |
| 检测率 | 准确识别事件的比例 | 95% | 85% | >95% |
| 误报率 | 误报/总告警比例 | 3% | 10% | <5% |
| 闭环率 | 完成复盘的事件占比 | 100% | 70% | >90% |

### 5. 复盘中常见问题清单

#### 技术层面
```text
□ 日志覆盖是否完整？（缺少哪些关键日志源？）
□ 告警规则是否准确？（产生了多少误报/漏报？）
□ EDR覆盖率是否达标？（有多少主机未安装？）
□ 补丁管理是否存在缺口？（被利用的漏洞何时发布补丁？）
□ 网络分段是否有效？（横向移动是否被阻止？）
□ 备份策略是否有效？（RTO/RPO是否满足？）
```

#### 流程层面
```text
□ 事件分级是否准确？（初判严重等级是否恰当？）
□ 上报流程是否通畅？（上报链路是否过长？）
□ 沟通渠道是否有效？（团队间沟通有无障碍？）
□ 决策流程是否高效？（管理层决策是否及时？）
□ 外部协作是否顺畅？（厂商、执法、PR响应速度？）
```

#### 人员层面
```text
□ 值班人员配置是否合理？（7x24覆盖？）
□ 技能培训是否到位？（工具操作熟练度？）
□ 复盘文化和氛围如何？（是否坦诚复盘的土壤？）
□ 交接班流程是否规范？（信息有无丢失？）
```

### 6. 复盘报告示例 — Executive Summary

```text
# 执行摘要

## 事件概述
2024年X月X日，我司检测到一起针对[系统名称]的勒索软件攻击事件。
攻击者通过[攻击方式]获得初始访问权限，利用[漏洞/手段]提升权限并部署勒索软件。
此次事件共影响[X]台服务器，加密约[X]个文件。

## 应急处置
IRT团队在告警后15分钟内完成事件确认和初步定级（P1），
30分钟内完成受影响主机网络隔离，
24小时内完成所有受感染系统的恶意软件清除，
48小时内完成业务恢复。

## 根因分析
1. 直接原因：[漏洞/弱口令/钓鱼]
2. 根本原因：缺乏[多因素认证/补丁管理/安全基线]

## 改进计划
1. [短期措施] — 修复安全配置漏洞
2. [中期措施] — 部署MFA
3. [长期措施] — 完善安全架构
```
